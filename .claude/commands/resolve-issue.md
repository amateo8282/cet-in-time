# GitHub 이슈 해결 계획 커맨드

GitHub 이슈 번호를 입력받아 이슈를 분석하고 해결 계획을 수립합니다.

## 입력

**이슈 번호**: $ARGUMENTS

## 수행 작업

### 1. 이슈 조회

`gh issue view $ARGUMENTS` 명령어를 사용하여 이슈 정보를 가져오세요:

- 제목
- 본문
- 라벨
- 담당자
- 상태
- 댓글 (`gh issue view $ARGUMENTS --comments`)

### 2. 이슈 분석

조회된 이슈를 분석하여 다음을 파악하세요:

- **문제 정의**: 해결해야 할 핵심 문제가 무엇인가?
- **범위**: 영향받는 기능, 모듈, 사용자는?
- **제약 조건**: 기술적/비즈니스적 제약이 있는가?
- **성공 기준**: 문제가 해결되었음을 어떻게 확인할 수 있는가?

### 3. 코드베이스 탐색

이슈와 관련된 코드베이스를 탐색하세요:

- 관련 파일 및 모듈 파악
- 기존 구현 패턴 확인
- 영향받는 코드 영역 식별
- 기존 테스트 확인

### 4. 해결 계획 수립

클린 아키텍처와 TDD 원칙에 따라 해결 계획을 수립하세요:

#### 계층별 태스크 분류

1. **Domain Layer (도메인 계층)**
   - 필요한 엔티티, 값 객체 수정/추가
   - 도메인 서비스 및 비즈니스 규칙

2. **Application Layer (애플리케이션 계층)**
   - 유스케이스 정의/수정
   - 필요한 인터페이스 변경

3. **Infrastructure Layer (인프라 계층)**
   - 외부 API, 데이터베이스 연동
   - Repository 구현 변경

4. **Presentation Layer (프레젠테이션 계층)**
   - UI 컴포넌트 변경
   - 상태 관리, 사용자 상호작용

#### TDD 사이클 연계

각 태스크에 대해 TDD 사이클(Red-Green-Refactor)을 적용하세요.

### 5. 출력 형식

다음 형식으로 해결 계획을 출력하세요:

```markdown
## 이슈 분석

**이슈 번호**: #$ARGUMENTS
**제목**: [이슈 제목]
**상태**: [Open/Closed]

### 문제 정의
[핵심 문제 설명]

### 범위
[영향받는 기능, 모듈]

### 성공 기준
- [기준 1]
- [기준 2]

---

## 관련 코드

| 파일 | 역할 | 변경 필요 |
|------|------|----------|
| [파일 경로] | [설명] | [예/아니오] |

---

## 해결 계획

### 태스크 1: [태스크 제목]

**계층**: [Domain/Application/Infrastructure/Presentation]

**설명**: [상세 설명]

**TDD 단계**:
1. Red: [실패하는 테스트]
2. Green: [최소 구현]
3. Refactor: [리팩토링 포인트]

**완료 조건**:
- [ ] [조건 1]
- [ ] [조건 2]

---

## 의존성 그래프

태스크 1 → 태스크 2 → 태스크 3

---

## 예상 커밋

1. `[커밋 메시지 1]`
2. `[커밋 메시지 2]`
```

### 6. 사용자 검토

계획을 제시한 후 다음을 확인하세요:

- 계획에 대한 피드백
- 수정/보완 사항
- 우선순위 조정 필요 여부
- 추가 고려사항

---

## 실행

위 지침에 따라 이슈 #$ARGUMENTS를 조회하고 분석하여 해결 계획을 수립한 후, 사용자의 검토를 받으세요.
